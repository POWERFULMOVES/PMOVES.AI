# Integration Plan: PMOVES v5.12 and Wger Fitness Manager

## 1\. Technical Architecture Compatibility

**Overview:** PMOVES v5.12 is a modular multi-agent AI system, while Wger is an API-centric fitness and nutrition platform. Their architectures are complementary ‚Äì PMOVES excels at orchestrating AI agents and managing knowledge, and Wger provides a structured data source for personal health metrics via a RESTful API. The integration will leverage PMOVES‚Äôs support systems (Archon, Hi-RAG, etc.) to ingest and reason over Wger data, treating Wger as an **external service within the PMOVES ecosystem**. This section outlines how the components align.

High-level architecture linking PMOVES and Wger components  
[\[1\]](file://file-KvyPf9XMVfpWHZvzP3CrW2#:~:text=subgraph%20,end)[\[2\]](file://file-Aht547Wckx9x4tbq88Qp2M#:~:text=%2A%2A6.3%20Self,Value%20Proposition)

*Diagram: PMOVES‚Äôs layered architecture (left, gray) interfacing with Wger‚Äôs API-first system (right, yellow). Arrows show data flow: Wger‚Äôs REST API supplies user fitness data to PMOVES‚Äôs knowledge base (Supabase vector DB) for AI agents to use; PMOVES workflows can also send updates back to Wger.*

**PMOVES Architecture:** At its core is **Agent Zero**, the orchestrator that delegates tasks to specialized modules[\[3\]](file://file-KvyPf9XMVfpWHZvzP3CrW2#:~:text=subgraph%20,Maker%20%26%20Orchestrator%22%5D%20end). Key support components include **Archon** (knowledge & agent management), **Hi-RAG** (Hierarchical Retrieval-Augmented Generation for deep reasoning), **LangExtract** (structured text extraction), and **n8n** (workflow automation)[\[4\]](file://file-KvyPf9XMVfpWHZvzP3CrW2#:~:text=subgraph%20,end). PMOVES uses **Supabase (Postgres)** as a unified database with vector search (pgvector) for knowledge storage[\[5\]](file://file-KvyPf9XMVfpWHZvzP3CrW2#:~:text=The%20Data%20%26%20Operational%20Backbones,privacy%20and%20efficient%20local%20processing). All components are containerized via Docker for easy deployment[\[6\]](file://file-KvyPf9XMVfpWHZvzP3CrW2#:~:text=subgraph%20,end), aligning with Wger‚Äôs Docker-based deployment model[\[7\]](file://file-Aht547Wckx9x4tbq88Qp2M#:~:text=Self,1). This ensures both systems can co-reside on the same infrastructure and network.

**Wger Architecture:** Wger is a Django-based application exposing a comprehensive **REST API** as its single source of truth[\[8\]](file://file-Aht547Wckx9x4tbq88Qp2M#:~:text=,into%20the%20wger%20REST%20API). It stores data (workout routines, nutrition plans, weight logs, etc.) in a relational database (typically PostgreSQL) and includes official web (React) and mobile (Flutter) clients that consume the API[\[9\]](file://file-Aht547Wckx9x4tbq88Qp2M#:~:text=The%20wger,publicly%20hosted%20web%20application)[\[10\]](file://file-Aht547Wckx9x4tbq88Qp2M#:~:text=1.%20,11). The API is organized into public and private endpoints, requiring token authentication for user-specific data[\[11\]](file://file-Aht547Wckx9x4tbq88Qp2M#:~:text=). Crucially, Wger is designed for **self-hosting** via Docker Compose, giving full data control and privacy[\[2\]](file://file-Aht547Wckx9x4tbq88Qp2M#:~:text=%2A%2A6.3%20Self,Value%20Proposition) ‚Äì a philosophy that matches PMOVES‚Äôs emphasis on local-first deployments and data sovereignty. Both systems favor a **modular, API-driven approach**, making integration feasible through standard HTTP calls and database connectors.

**Compatibility Considerations:** The integration will treat Wger as a data provider service within PMOVES: \- **Data Access:** PMOVES agents can fetch Wger data through the Wger REST API. The API-first design of Wger means PMOVES doesn‚Äôt need to directly manipulate Wger‚Äôs database; it can use high-level endpoints (e.g. /api/v2/workoutlog/, /api/v2/nutrition/) to retrieve information[\[12\]](file://file-Aht547Wckx9x4tbq88Qp2M#:~:text=,Practical%20Examples). This aligns with PMOVES‚Äôs multi-agent approach where an agent (or n8n workflow) can call external APIs as a tool. \- **Networking:** Running Wger in the same Docker network as PMOVES allows secure, local API calls (e.g. http://wger:8000/api/v2/...). There are no fundamental port or protocol conflicts ‚Äì Wger‚Äôs web service listens on configurable ports, and PMOVES services (Supabase, n8n, etc.) have their own ports. Reverse proxies or internal DNS can be set so that PMOVES components can refer to the Wger service by name. \- **Data Format:** Wger‚Äôs API returns JSON structured data (workouts, meals, logs). PMOVES‚Äôs LangExtract can convert JSON or text into embedding-friendly formats (JSONL)[\[13\]](file://file-TZV4MCWgtaiHBW5mUiTZJ4#:~:text=Service%20,d%20langextract). The structured nature of Wger‚Äôs data (dates, exercise names, numeric values) can be annotated or summarized into text for indexing by PMOVES‚Äôs vector database. \- **Authentication:** Wger uses token-based auth for private data[\[14\]](file://file-Aht547Wckx9x4tbq88Qp2M#:~:text=The%20API%20employs%20a%20clear,in%20the%20request%20header). PMOVES can securely store the API token (e.g. in Supabase or environment) and include it in requests. If multiple user accounts are used (e.g. one for each persona or patient), PMOVES can manage multiple tokens or an OAuth flow. The integration plan includes an **auth adapter** to handle obtaining and refreshing tokens, similar to other integrations[\[15\]](https://drive.google.com/file/d/1Id3h_AbSrUmNIRw6fjucQhIyEP26rsEJ). \- **Modularity:** Neither system requires invasive changes to integrate. Wger‚Äôs API allows non-invasive data access. PMOVES‚Äôs architecture is explicitly built to integrate external tools (e.g. Firefly III is already integrated as a finance component[\[16\]](file://file-KvyPf9XMVfpWHZvzP3CrW2#:~:text=subgraph%20,Models%3A%20Ollama%2C%20NVIDIA%20NIM%2C%20Nemo)[\[17\]](https://drive.google.com/file/d/1Id3h_AbSrUmNIRw6fjucQhIyEP26rsEJ)). We will add Wger as a new module in this ecosystem, preserving the decoupling ‚Äì Wger remains an independent service that PMOVES can replace or turn off without breaking other functions.

In summary, the API-driven, containerized natures of PMOVES and Wger make them architecturally compatible. Wger will function as a **well-contained ‚Äúhealth data provider‚Äù** for PMOVES, and PMOVES will enrich Wger‚Äôs data with AI-driven insights and cross-domain connections (e.g. linking fitness with finance). Both systems emphasize local deployment and privacy, ensuring the combined architecture remains self-hosted and secure.

## 2\. Integration Objectives and Approach

The integration aims to achieve several key objectives:

* **2.1 Ingesting and Indexing Wger User Data:** Enable PMOVES to **ingest, index, and reason over personal fitness/nutrition data** from Wger. Using PMOVES‚Äôs ingestion pipeline, Wger data (workout routines, meal plans, body weight logs, etc.) will be pulled into the PMOVES knowledge base for analysis. This involves retrieving data via the Wger API and storing representations in Supabase (with pgvector) so that Hi-RAG can perform semantic search and retrieval on it[\[5\]](file://file-KvyPf9XMVfpWHZvzP3CrW2#:~:text=The%20Data%20%26%20Operational%20Backbones,privacy%20and%20efficient%20local%20processing). Each relevant piece of Wger data becomes a **document or ‚Äúchunk‚Äù** in the PMOVES knowledge base, tagged with metadata (e.g. user, date, type of entry). For example, a weekly workout routine might be stored as a document with sections for each day‚Äôs exercises, and a weight log entry might be a short text snippet like ‚Äú**2025-09-29:** Weight 160 lb‚Äù indexed for trend analysis. PMOVES v5.12‚Äôs new **grounding pack** structure will be used to keep this data organized (see 2.2). The ingestion process can be automated via **n8n workflows** or a custom sync script:

* *Scheduled Pull:* On a schedule (e.g. nightly or hourly), n8n triggers a workflow to call Wger‚Äôs API endpoints for each user‚Äôs latest data (only pulling new or updated entries). The results (JSON) are then fed to LangExtract or a formatting step to produce text suitable for embedding. Using the documents/sections/chunks schema in Supabase, the data is upserted along with vector embeddings[\[18\]](file://file-YQJaTqSeB8FgAyEzqVtP9V#:~:text=,adjust%20dim)[\[19\]](file://file-YQJaTqSeB8FgAyEzqVtP9V#:~:text=create%20index%20if%20not%20exists,lists%20%3D%20100). This yields semantic indexing of Wger data, allowing PMOVES agents to retrieve information like ‚Äúmeals logged this week‚Äù or ‚Äúlast workout details‚Äù via vector search.

* *Real-time Ingestion:* Optionally, integrate more closely by using Wger‚Äôs Django signals or a lightweight webhook mechanism. For example, when a user logs a new weight or completes a workout on Wger, a custom **webhook** could POST that event to a PMOVES endpoint. PMOVES‚Äôs ingestion service (or an n8n webhook node) would receive the event and immediately index the new data. This ‚Äúon-event‚Äù sync complements the scheduled pulls, ensuring the AI is up-to-date. The PMOVES technical blueprint suggests placing such hooks in an integrations/sync.py module[\[15\]](https://drive.google.com/file/d/1Id3h_AbSrUmNIRw6fjucQhIyEP26rsEJ) ‚Äì this module can listen for Wger events or poll diff endpoints to keep data in sync.

* **2.2 Bi-Directional Sync and Contextual Grounding:** Establish a **bi-directional link** so that PMOVES not only consumes Wger data but can also push or utilize updates in Wger. Bi-directional integration means changes made via the AI side can reflect back in Wger, maintaining a single source of truth for fitness records. For instance, if an AI persona (or the user via PMOVES) adjusts a workout routine or adds a note, PMOVES can call Wger‚Äôs API to update the official record (e.g. create a new workout entry or add a note in Wger‚Äôs database). This ensures Wger remains the authoritative store for health data, and avoids forking data.

To support rich AI reasoning, the Wger data will be embedded *‚Äúin context‚Äù* for PMOVES‚Äôs agents through **grounding packs**: \- **Grounding Packs for Wger Data:** Grounding packs are a new concept in PMOVES v5.12 that group relevant knowledge chunks for a persona[\[20\]](file://file-YQJaTqSeB8FgAyEzqVtP9V#:~:text=A%20tight%2C%20copy%E2%80%91pasteable%20upgrade%20plan,contracts%2C%20and%20minimal%20API%20specs)[\[21\]](file://file-YQJaTqSeB8FgAyEzqVtP9V#:~:text=,%40handle%20or%20team). We will create one or more packs that represent the user‚Äôs Wger data. For example, a pack named "fitness-journal@1.0" could contain all of a user‚Äôs workout logs, and "meal-plans@1.0" could contain nutrition logs. Each pack is versioned and can be updated as new data comes in. PMOVES‚Äôs retrieval is pack-scoped, meaning an agent can be instructed to search within a particular pack for relevant info[\[22\]](file://file-YQJaTqSeB8FgAyEzqVtP9V#:~:text=,a%20gate%20for%20persona%20publish). By associating the Wger data pack with the persona (see 2.3), every query that persona handles will automatically consider that user‚Äôs latest health data. This provides *contextual embedding* ‚Äì the persona‚Äôs responses will be grounded in the factual data from Wger (e.g., it can cite the user‚Äôs recent weight trend or meals). \- **Sync Back to Wger:** For the ‚Äúpush‚Äù direction, PMOVES will use Wger‚Äôs REST API to propagate changes. This could be triggered by user actions or AI suggestions. For example, if the persona AI suggests a new workout routine and the user approves, PMOVES can call POST /api/v2/workout/ to create it in Wger. Similarly, if a persona logs a meal or weight on behalf of the user (perhaps via voice command), it will use Wger‚Äôs API to record it so that Wger‚Äôs database stays updated. Automating this through n8n ensures consistency ‚Äì an n8n workflow can listen for a new ‚Äúvirtual log‚Äù entry in Supabase and then call Wger API to mirror it. \- **Conflict Resolution:** In a bi-directional setup, it‚Äôs important to avoid conflicts (e.g., the same data being added twice). The integration will define a source of truth and timestamps. Wger‚Äôs entries have timestamps; PMOVES can maintain a last-synced timestamp to fetch only new items. For pushes from PMOVES, unique IDs from Wger can be stored in the Supabase metadata to ensure no duplicate writes. Because PMOVES ultimately relies on Wger as authority, we‚Äôll favor a strategy where Wger IDs are keys for data in the vector index (i.e., each chunk in Supabase is tagged with the Wger entry ID), enabling updates in place if Wger data changes. \- **Performance:** Both systems handle moderate data volumes (Wger logs and plans are relatively small text). Indexing with pgvector is efficient and can handle the additional records. The retrieval pipeline (Hi-RAG) can combine **vector similarity and keyword search** on these packs[\[22\]](file://file-YQJaTqSeB8FgAyEzqVtP9V#:~:text=,a%20gate%20for%20persona%20publish), ensuring that even a large history of logs can be searched quickly. We will use batch ingestion (grouping multiple entries in one JSONL load) to speed up initial sync. Subsequent incremental updates are lightweight (often just a few new log entries per day).

* **2.3 Wellness Coaching via Grounded Personas:** Leverage the integrated data to **extend the PMOVES AI pipeline for personalized wellness coaching**. The vision is to deploy intelligent personas that use the combined knowledge of Wger (health data) and other domains (e.g. finance, general medical knowledge) to provide tailored guidance. We outline two persona-driven use cases:

* **Persona 1 ‚Äì Nurse Practitioner (Private Health Advisor):** A persona is created for the user‚Äôs mother, who is a nurse practitioner, to help her interact with patient fitness data in a secure, HIPAA-compliant manner. This persona acts as a *medical assistant agent* that can summarize and highlight patient information drawn from Wger. In practice:

  * Each patient under the mother‚Äôs care can have a profile in Wger (e.g., a separate Wger user account or tagged dataset for each patient, since Wger is multi-user). The integration would ingest each patient‚Äôs Wger logs into separate grounding packs (e.g. patient\_john\_doe@1.0). The **owner field** in the grounding pack metadata will tie it to the nurse practitioner[\[23\]](file://file-YQJaTqSeB8FgAyEzqVtP9V#:~:text=create%20table%20if%20not%20exists,), and Supabase row-level security can enforce that only her persona can access those packs (see Privacy section).

  * The nurse practitioner persona is configured to load the relevant patient pack when queried. For example, when the nurse selects or asks about ‚ÄúJohn Doe‚Äôs routine,‚Äù the persona‚Äôs query is scoped to John‚Äôs pack, retrieving his latest exercise compliance, diet, and any notes.

  * **Review & Annotation Workflow:** Through a secure UI (possibly Archon‚Äôs web interface or a custom dashboard), the nurse can review a patient‚Äôs weekly routine and logs. The persona can present a summary: *‚ÄúJohn completed 3 of 4 prescribed workouts this week and his weight decreased by 2 lbs. He reported knee pain on Wednesday.‚Äù* The nurse can ask follow-up questions in natural language, like ‚ÄúWhich days did John report pain and what was he doing?‚Äù The persona will ground the answer in John‚Äôs logs[\[24\]](file://file-Aht547Wckx9x4tbq88Qp2M#:~:text=basic%20gym%20management%20functionalities,1)[\[25\]](file://file-Aht547Wckx9x4tbq88Qp2M#:~:text=A%20major%20feature%20is%20the,7), citing the exact entries (e.g. a note in Wger on Wednesday‚Äôs workout).

  * The nurse can then annotate or adjust the plan. For instance, she might say, *‚ÄúReduce John‚Äôs running intensity next week and add a rest day on Wednesday.‚Äù* The persona will interpret this and execute an update: it could modify the routine in Wger via API (lower running distance, insert rest day) so that John‚Äôs plan is updated. Alternatively, the persona could log an annotation (e.g., create a note attached to the Wednesday workout entry like ‚Äúadvised rest due to knee pain‚Äù). Wger‚Äôs API or an extended schema can store these notes, or PMOVES can maintain them in Supabase with references to the Wger routine (allowing private notes that patients don‚Äôt see).

  * **HIPAA Compliance:** This use-case keeps all patient data on a local server; PMOVES‚Äôs local LLMs analyze it without transmitting data externally. The persona‚Äôs responses can be designed to include sources (e.g., referencing the date of a log) for auditability. Every access the persona makes to patient data can be logged (via n8n or Supabase logging tables) to create an audit trail ‚Äì a common requirement under HIPAA. We ensure that **only the nurse‚Äôs credentials/persona can access those packs**, so even if multiple personas exist, patient data packs are isolated (see Privacy & Compliance in Section 3 for technical enforcement).

* **Persona 2 ‚Äì Family Member (Personal Wellness & Finance Coach):** Another persona is configured to assist the user‚Äôs daughter in optimizing her fitness and finances together. This persona acts as a *holistic life coach*, combining data from **Wger (fitness/nutrition)** and **Firefly III (personal finance)**. By integrating both data sources, the agent can provide advice that accounts for both health and budget considerations.

  * **Integrated Data Grounding:** The daughter‚Äôs Wger data (workout schedules, diet logs, weight trends) will be available in a fitness pack, while her Firefly III data (spending categories, budgets, transactions) is available in a finance pack. PMOVES already integrates Firefly III (financial records can be pulled via its API and indexed)[\[17\]](https://drive.google.com/file/d/1Id3h_AbSrUmNIRw6fjucQhIyEP26rsEJ). The persona can be set with *two default packs*: e.g., my\_fitness@1.0 and my\_finance@1.0, so it always considers both health and finance context when answering.

  * **Use Case ‚Äì Daily Guidance:** Each day, the persona could generate a personalized summary or ‚Äúnudge‚Äù. For example: *‚ÄúGood morning\! You have a strength training workout planned today. You‚Äôve been consistent 3/5 days this week ‚Äì great job\! üìà Also, you are $50 under your weekly food budget so far. Perhaps use some of that to buy fresh veggies for meal prep, which will help your diet and still save money. ü•¶üí∞‚Äù* This message combines knowledge from Wger (workout consistency, diet) and Firefly (budget status). We can implement this via an n8n scheduled workflow: at a set time, the workflow fetches the latest data from both systems, composes a prompt for the LLM (Agent Zero or a dedicated prompt template for the persona), and then delivers the generated nudge via a communication channel (could be a push notification, an email, or displayed in the PMOVES UI). Agent Zero can also generate such summaries on demand if the daughter asks, ‚ÄúHow am I doing on workouts and spending this week?‚Äù ‚Äì it will retrieve key facts (e.g. ‚Äú4 workouts done, $100 spent on restaurants‚Äù) and then the persona will respond conversationally with encouragement or advice.

  * **Contextual Help:** The persona can answer questions like, *‚ÄúCan I afford a new gym membership?‚Äù* by looking at financial data (monthly budget surplus) and fitness data (current exercise routine). If Firefly shows, say, $30/month available in her recreation budget, and Wger data suggests she‚Äôs been lacking gym equipment, the persona might advise a basic gym membership that fits that budget.

  * **Workflow Automation:** Some actions can be automated across domains. For instance, if the daughter logs a meal in Wger that exceeds her calorie goal, the persona (via n8n) could flag a corresponding entry in Firefly (like spending on takeout) and maybe suggest reallocating funds from dining-out budget to groceries for cooking healthier meals. Conversely, if she overspends in Firefly on restaurants, the persona can gently suggest checking her Wger meal logs and perhaps planning home-cooked meals (and even use Wger‚Äôs nutrition database to suggest a recipe within her calorie targets).

  * **User Experience:** The daughter interacts with the persona likely through a chat interface (e.g., Agent Zero‚Äôs terminal or a simple web chat). This persona will be friendly and supportive in tone. Using PMOVES‚Äôs **local LMs** ensures privacy ‚Äì her financial and fitness data remain on the device. The persona can display charts or tables for clarity: e.g., a table of weekly progress (workout days vs. spending) or even call external tools for visualization. Since PMOVES has tools like ComfyUI and plotting libraries, a future enhancement could generate a progress chart image to send her. However, even in text, the agent can enumerate key metrics clearly, keeping messages short and actionable.

By achieving these objectives, PMOVES becomes a unified intelligence hub for personal wellness. It brings Wger‚Äôs strength (fitness tracking) into the fold of PMOVES‚Äôs multi-agent reasoning, allowing richer insights that neither system could provide alone. The next sections detail the privacy safeguards, deployment steps, and design considerations to implement this plan.

## 3\. Privacy and Compliance (HIPAA and Data Security)

Integration must be done with stringent privacy and security measures, especially since one scenario involves personal health information (potentially PHI under HIPAA). We outline how **local-first data control, encryption, and role-based security** are enforced:

* **Self-Hosted, Local-First Deployment:** The entire PMOVES \+ Wger stack will be deployed in a self-hosted environment (e.g., on-premises server or a secure private cloud) under the user‚Äôs control. Wger‚Äôs open-source nature allows running it independently from the public wger.de service, meaning no fitness data needs to leave the local server[\[2\]](file://file-Aht547Wckx9x4tbq88Qp2M#:~:text=%2A%2A6.3%20Self,Value%20Proposition). PMOVES is designed for local operation with optional internet services ‚Äì for HIPAA compliance, any internet-dependent features (like calling external AI APIs) will be disabled or pointed to compliant local alternatives. All AI inference can be done with **local models** (Ollama, NVIDIA NeMo, etc., as indicated in the architecture[\[5\]](file://file-KvyPf9XMVfpWHZvzP3CrW2#:~:text=The%20Data%20%26%20Operational%20Backbones,privacy%20and%20efficient%20local%20processing)) to avoid transmitting sensitive data to third parties. This local-first approach addresses the primary HIPAA requirement of maintaining control over PHI and not sharing it without authorization.

* **Encryption at Rest and in Transit:** Supabase (Postgres) will store embedded Wger data and any additional notes. We will enable database-level encryption where possible. For example, if using a self-hosted Postgres 15+, we can employ **Transparent Data Encryption** plugins or ensure the underlying disk volume is encrypted. All Docker volumes for databases should reside on encrypted filesystems. Wger‚Äôs own database similarly should be on encrypted storage. In transit, communications between components (e.g., PMOVES to Wger API) occur over the Docker network ‚Äì we will enforce TLS for any external access and optionally within the internal network (though if it‚Äôs a closed Docker network on one host, it may be sufficient to rely on that isolation). Supabase‚Äôs API can be served over HTTPS (in development it‚Äôs http on localhost, but for production behind a reverse proxy with TLS). Any remote access (e.g., if the nurse connects from a clinic PC to her home server) should be done via VPN or secure channels like HTTPS with client certs.

* **Access Control and Isolation:** We leverage **Supabase‚Äôs Role-Based Access Control and Row-Level Security (RLS)** to partition data. In the PMOVES v5.12 schema, new tables such as grounding\_packs, pack\_members, and personas have an owner or similar fields[\[23\]](file://file-YQJaTqSeB8FgAyEzqVtP9V#:~:text=create%20table%20if%20not%20exists,) which can represent the user or persona that owns that data pack. We will enable RLS policies on these tables so that, for example, the nurse practitioner can only query packs that are designated for her (and/or marked as containing patient data she‚Äôs allowed to see), and the daughter‚Äôs persona only sees her own packs. A simple RLS policy might match the pack‚Äôs owner to the current Supabase auth role/user. Supabase‚Äôs GoTrue service (enabled in the full stack[\[26\]](file://file-HoYko7qUkFv7vbhMMtGs2q#:~:text=Option%20A%20%E2%80%94%20Supabase%20CLI,not%20necessary%20for%20local)) can manage distinct user accounts for ‚Äúnurse‚Äù and ‚Äúdaughter,‚Äù each mapped to certain roles. For instance, a role nurse\_role can be given select access to patient data tables, whereas daughter\_role has access to her fitness and finance data only. We will *restrict write access* to the integration services (via a service role API key) so that end-users cannot maliciously write to tables except through the sanctioned workflows[\[27\]](file://file-YQJaTqSeB8FgAyEzqVtP9V#:~:text=,assets%20for%20select%20using%20%28true). These measures ensure **principle of least privilege** ‚Äì even if both personas run on one system, they cannot read each other‚Äôs data.

* **HIPAA-Specific Measures:** In a HIPAA context, beyond encryption and access control, we need auditability and safeguarding of PHI. We will implement:

* **Audit Logs:** Every query or action the nurse‚Äôs persona performs on patient data can generate an entry in an audit table (with timestamp, patient ID, action, etc.). Supabase‚Äôs Realtime or triggers can log SELECTs on sensitive tables, or simpler, we log at the application level in n8n each time a patient‚Äôs pack is accessed. This provides an audit trail for compliance review.

* **De-identification Option:** If the nurse‚Äôs use case involves real patient data, we might consider de-identifying data before it enters the AI pipeline. For example, patient names could be replaced with codes in the AI‚Äôs context (since the nurse knows which patient she queried anyway). This way, if any logging or debugging occurs, it minimizes exposure of identifiers.

* **No Third-Party Data Leakage:** We will ensure that the AI personas do not inadvertently share PHI in channels that are not secure. For example, if using Discord for notifications (as PMOVES can integrate with Discord for certain alerts[\[28\]](file://file-YQJaTqSeB8FgAyEzqVtP9V#:~:text=Publisher%20PUBLISHER_NOTIFY_DISCORD_WEBHOOK%3D%20https%3A%2F%2Fdiscord,JELLYFIN_LIBRARY_ID)), we will disable any such notifications for content containing PHI. Similarly, any cloud backup of the vector DB must be encrypted or avoided. The system will run entirely on infrastructure under a Business Associate Agreement (if cloud) or on-prem hardware to satisfy HIPAA rules.

* **Compliance of Wger and Firefly:** Both Wger and Firefly III are self-hosted solutions. Wger is not HIPAA-certified out-of-the-box, but by self-hosting and limiting access, we can make it part of a HIPAA-compliant setup. We ensure that any ePHI (e.g., health logs) in Wger is accessible only to authorized persons (the nurse) via Wger‚Äôs own user management. Firefly III holds financial data, which is sensitive but not PHI; however, we treat it with similar care (financial privacy). Access to Firefly‚Äôs API will similarly use secure tokens and limited roles.

* **Testing and Hardening:** Before deployment, we will run penetration tests within the integrated environment. We will enable logging at the network level (possibly using a proxy to log all API calls between PMOVES and Wger) to monitor that no data is leaking to unexpected destinations. Supabase‚Äôs studio and monitoring tools can help ensure RLS policies are working as intended (e.g., try queries as different roles). For production, we favor using the full Supabase stack (with GoTrue for auth, etc.) as described in PMOVES docs[\[26\]](file://file-HoYko7qUkFv7vbhMMtGs2q#:~:text=Option%20A%20%E2%80%94%20Supabase%20CLI,not%20necessary%20for%20local) rather than the dev stub, to utilize its security features. The PMOVES docs note that the provided Compose setup is simplified and not hardened[\[29\]](file://file-HoYko7qUkFv7vbhMMtGs2q#:~:text=Notes%20,Supabase%20CLI%20in%20%E2%80%9Cprod%E2%80%9D%20mode); accordingly we will apply the necessary hardening (strong JWT secrets, secure storage backend, etc.) for the deployment.

In essence, **data privacy is central** to the integration. By keeping everything self-hosted and using strong access controls, we align with both the philosophy of Wger (data sovereignty for users[\[2\]](file://file-Aht547Wckx9x4tbq88Qp2M#:~:text=%2A%2A6.3%20Self,Value%20Proposition)) and PMOVES (local control over data and models[\[30\]](file://file-KvyPf9XMVfpWHZvzP3CrW2#:~:text=The%20PMOVES%20,aspects%20of%20the%20PMOVES%20system)). The result is an environment where health and finance data is used to empower intelligent coaching personas **without compromising privacy or compliance**.

## 4\. Deployment Plan and DevOps Integration

Deploying the combined system will involve orchestrating multiple services (PMOVES components, Wger, and supporting services like databases) in a unified manner. We will use **Docker Compose and PMOVES‚Äôs Makefile targets** to automate this. Below is a step-by-step deployment plan:

**4.1 Environment Configuration:** \- Begin by configuring environment variables for all services. PMOVES has an .env (or several .env files) that will need entries for Wger integration. If the Wger instance requires a login to obtain a token, you‚Äôll create a user (or multiple users) in Wger first (e.g., an admin account for the nurse, and separate account for the daughter). ## 5\. UX and Workflow Design

With the technical integration in place, designing a clear user experience (UX) is crucial. We focus on how the two personas ‚Äì the nurse practitioner and the family member (daughter) ‚Äì will interact with the system, and how the workflows feel natural and intuitive. ### 5.1 Nurse Practitioner Workflow

**Accessing Patient Data:** The nurse practitioner (NP) will likely interact through a secure web interface or chat console. We can use **Archon‚Äôs web UI** for knowledge and task management[\[1\]](file://file-KvyPf9XMVfpWHZvzP3CrW2#:~:text=subgraph%20,end), adapted for clinical use. On logging in (as ‚ÄúNP‚Äù user), she is presented with a list of her patients or a search bar to query patient data. This might trigger the persona to greet the nurse with a summary: e.g., *‚ÄúYou are now viewing* *John Doe‚Äôs fitness profile. Last week: 2/3 workouts completed, weight stable at 180 lb.‚Äù*

**Reviewing Routines and Logs:** The nurse can click on sections like ‚ÄúWorkouts‚Äù, ‚ÄúNutrition‚Äù, or ‚ÄúProgress‚Äù to see the raw data from Wger in a human-readable form (possibly drawn via Wger‚Äôs API or via data already stored). This UI could embed components from Wger‚Äôs frontend (if we want, we could even link to the Wger web UI for detailed editing, but ideally keep the NP in one interface).

**Chat-Based Q\&A:** A highlight of the UX is the ability for the NP to ask natural language questions in a chat panel with the AI persona (let‚Äôs call it ‚ÄúClara, the clinical assistant‚Äù). Clara is a grounded persona with knowledge packs: the patient‚Äôs Wger data, possibly a medical knowledge base pack (for general guidelines), and any notes the nurse has added. Example interactions: \- NP: *‚ÄúClara, how has John‚Äôs blood pressure been doing during his exercises?‚Äù*

Notably, on 2025-09-20, it spiked to 140/90 during a high-intensity workout[\[24\]](file://file-Aht547Wckx9x4tbq88Qp2M#:~:text=basic%20gym%20management%20functionalities,1). Clara might generate a summary: *‚ÄúIn the last 4 weeks, John‚Äôs weight went from 182 lb to 180 lb (down 2 lb). He completed 10 out of 12 planned workouts (83%). The weeks of Sep 1 and Sep 15 he missed one session each.‚Äù* This uses the vector search to find weight entries and workout logs, then presents a coherent summary with some statistics ‚Äì something Hi-RAG can do by retrieving data and using an LLM to summarize.

The NP can drill down: *‚ÄúShow me the details for the week he missed sessions.‚Äù* Clara can then fetch that specific week‚Äôs log and either display a table or describe it day by day.

**Annotating and Adjusting Plans:** The system should allow the NP to record her insights or decisions. For example, if she identifies that John‚Äôs knee pain correlates with running, she might click ‚ÄúAdd Note‚Äù or just tell Clara: *‚ÄúNote that John experienced knee pain after running. Advise to replace one running session with cycling.‚Äù* Clara will confirm and either: \- Add a note in Wger for John (via API, e.g., Wger might not have a direct note field for a user, but perhaps using the ‚Äúnotes‚Äù section or as a comment in the workout log). \- Or update John‚Äôs routine in Wger: find the running exercise in the routine and mark it substituted with cycling (through the API or prompt the NP to confirm the change). \- Clara then replies, *‚ÄúNoted. **HIPAA/Privacy UX:** Because this involves PHI, the UI should always indicate the data is sensitive. Possibly have a banner ‚ÄúPatient Data ‚Äì Confidential‚Äù. Session timeouts should be enforced (auto-logout after inactivity). When the NP switches patients, ensure no data from the previous patient lingers in the view or context (the persona should drop the old pack and load the new one).

**Training and Calibration:** We may include a feature for the NP to give feedback on Clara‚Äôs advice. For instance, if an AI suggestion is incorrect or inappropriate, the NP can flag it (maybe a ‚Äúthumbs down‚Äù button). This feedback can be logged to refine the persona or at least recorded for accountability. In deployment, we might keep the AI responses relatively conservative (it should cite data often and refrain from giving medical advice beyond what the data shows, unless we also integrate a medical knowledge base thoroughly).

**Wrap-up:** The NP‚Äôs overall experience should be that **Clara (AI)** streamlines her review process: instead of manually combing through logs, she gets quick answers and can make updates through conversation. Yet, the NP remains in control ‚Äì she decides what changes to accept and can always cross-verify with the raw data (which is presented or accessible). The integration with Wger means if she ever doubts the AI, she can open the Wger interface itself for a sanity check, but ideally the AI‚Äôs answers are grounded and referenceable, building trust over time.

The persona can generate it on request (though long text responses should be used sparingly). \- **Charts/Diagrams:** With integration to Jellyfin or ComfyUI (discussed later), we might even produce a chart image of her progress. For now, textual description and maybe ASCII art (like a progress bar made of characters) can be enough. \- **Persona Personality:** The coach persona should maintain a motivational, empathetic tone. Possibly include emojis for encouragement (as seen in examples) to make the experience more fun and less sterile.

**Privacy:** On the daughter‚Äôs side, privacy is about ensuring her data is not seen by others. That‚Äôs covered by accounts and RLS as discussed. But from a UX perspective, ensure she knows her data is stored locally (perhaps a note like ‚ÄúAll your data stays on this device ‚Äì private to you‚Äù). This can increase her trust in the system to share honest info.

**Ease of Use:** The daughter‚Äôs workflow should require minimal tech know-how. She doesn‚Äôt need to know about Supabase or PMOVES internals. She just knows she has a personal AI app that connects to her two self-hosted apps (Wger and Firefly). We can unify login if possible: maybe the same credentials log her into the PMOVES interface and those credentials could also be used for Wger‚Äôs app and Firefly if we set up single sign-on. (The technical report even suggests an OAuth SSO integration with Odoo, similar could be done with Supabase as an auth provider for Firefly and Wger[\[45\]](https://drive.google.com/file/d/1Id3h_AbSrUmNIRw6fjucQhIyEP26rsEJ), but that‚Äôs advanced). Even without SSO, since she‚Äôs likely the sole user of her system, we can simplify authentication flows (perhaps auto-login her persona on device startup, etc., assuming device is secure).

**Conclusion of UX:** The aim is that interacting with the integrated system feels like interacting with a **knowledgeable personal coach** who is aware of both your physical activities and your financial constraints. By grounding the AI in actual data (from Wger and Firefly), the advice is personalized and actionable. The user interfaces ‚Äì whether it‚Äôs chat, dashboards, or notifications ‚Äì should reinforce that the AI‚Äôs suggestions are coming from her own life‚Äôs data (e.g., citing ‚Äúyour budget‚Äù or ‚Äúyour log on Tue‚Äù). This fosters trust and makes the suggestions more impactful.

## 6\. Optional Enhancements and Future Integrations

Finally, we consider some optional enhancements that could further enrich the PMOVES-Wger integration, looking at wearable devices, third-party trackers, and AI analysis of media (photos/videos). These are forward-looking ideas to keep in mind:

* **6.1 Wearable Device Integration:** Incorporating data from fitness trackers (Fitbit, Apple Watch, Garmin, etc.) can provide real-time activity and health metrics to PMOVES. For example, step counts, heart rate, sleep quality, and calories burned are all valuable context for a wellness persona.

* *Approach:* Rather than building direct integrations with each wearable API from scratch, we could use an intermediary or the wearable‚Äôs API if accessible. Fitbit, for instance, has an OAuth-based API to fetch daily steps, heart rate, and even sleep logs. Apple Health data could be exported via HealthKit to a file or synced through iCloud (less straightforward on self-hosting). We might consider using an open-source aggregator like **Gadgetbridge** (for local syncing of devices) or simply allow the user to periodically import data (e.g., upload a Apple Health export).

* *Integration into Wger:* One strategy is to feed wearable data into Wger if Wger has placeholders for it. Wger doesn‚Äôt natively track steps or sleep, but it does track weight and body metrics. We could extend Wger‚Äôs database (since it‚Äôs self-hosted, schema can be modified) to have a ‚Äúdaily steps‚Äù field or a separate table for general health metrics. Alternatively, store this data in Supabase directly (as part of PMOVES knowledge base). The persona can still access it from Supabase.

* *Use Cases:* If a user‚Äôs Fitbit shows they consistently burn 500 calories through exercise daily, the persona can factor that into nutrition advice. Or if the wearable shows poor sleep for a few nights, the persona (especially for the daughter) might suggest taking it easy on intense workouts or adjust her schedule.

* *Privacy:* Wearable data can be sensitive (heart rate, location if any). Ensure any integration fetches goes through secure channels (the user likely has to provide API keys or tokens, which we store securely). All data stays local once fetched.

* The Wger analysis noted that integration with popular fitness trackers like Fitbit was a recognized opportunity in the community[\[46\]](file://file-Aht547Wckx9x4tbq88Qp2M#:~:text=could%20significantly%20expand%20the%20ecosystem%27s,17). By adding this, our system could become one of the more advanced community integrations of Wger, showcasing the power of combining data sources.

* **6.2 Analysis of Meal Photos:** Many users take photos of their meals rather than logging every ingredient. An AI vision model could analyze a meal photo to estimate its content (e.g., detect food items) and portion size to log approximate calories or nutritional info.

* *Jellyfin AI Stack:* Since the user has a Jellyfin media server with an AI plugin (the mention of Jellyfin AI implies they might be using something like Jellyfin with TensorFlow or DeepStack for object recognition), we can utilize that. For instance, Jellyfin could be pointed to a folder of ‚Äúmeal photos‚Äù. When the user uploads a photo (perhaps via a Nextcloud or a phone sync), Jellyfin‚Äôs AI processes it and could output tags (like ‚Äúpasta, tomato sauce, cheese‚Äù). We can take those tags and use a database of foods to estimate calories (Wger‚Äôs food database has millions of entries[\[47\]](file://file-Aht547Wckx9x4tbq88Qp2M#:~:text=exercises); perhaps we find the closest match via a search API).

* *LangExtract & OCR:* If the photo includes text (like a nutrition label or a restaurant receipt), LangExtract can OCR the text and extract structured data (like total calories, or ingredients). This could be combined with image recognition for better accuracy.

* *Workflow:* The user could simply say ‚ÄúI‚Äôm eating this‚Äù and snap a picture. The system (via a script or n8n) takes the image, sends it to an AI model (either a local one like YOLO for object detection or a specialized food model), gets back predictions. Then it could log a meal in Wger: e.g., if it detects ‚Äúburger and fries‚Äù, log it as an approximate entry (Wger‚Äôs database likely has generic entries for ‚Äúhamburger‚Äù etc.). The persona can respond with an estimate: *‚ÄúLogged your meal: roughly 800 kcal for a burger and fries. (Estimate)‚Äù*.

* *Benefits:* This lowers the barrier to logging nutrition, making the integrated coach more seamless. It also gives the persona more insight (e.g., if the photo shows a green salad vs a pizza, even without exact cals, the persona can adjust its advice qualitatively).

* *Technical Implementation:* We might integrate with **ComfyUI** or similar to run a pre-trained image classification model for food. Another route is to use cloud AI (like Google Vision has a Food ML model), but that conflicts with our privacy-first approach unless self-hosted or through a compliant route. There are some open datasets/models for food recognition (like Food-101). We could start with simple classification for major categories (vegetables, meats, etc.). Jellyfin‚Äôs AI plugin might identify general food items but not portion sizes, so this is an area for iterative improvement.

* **6.3 Analysis of Workout Videos:** For the truly ambitious, analyzing recorded workouts for form or rep counting could elevate the coaching experience.

* *Use Case:* The daughter sets up her phone to record her doing a workout. The video is saved (perhaps into a Jellyfin library or a designated folder). The AI processes it to detect exercises and count repetitions or check form (e.g., angle of limbs for a squat).

* *Tools:* OpenPose or MediaPipe (Pose estimation libraries) could be used to get keypoints from the video frames. With those, one could determine if she‚Äôs doing the exercise correctly (for example, depth of squat, back posture). Counting reps can be done by detecting cyclic motion of keypoints (for pushups, arms extending and bending).

* *Integration:* After analysis, the system could provide feedback: ‚ÄúI analyzed your squat video ‚Äì it looks like you did 10 reps, but your back was rounding on the last two. Be careful to keep it straight to avoid injury.‚Äù This feedback could be delivered by the persona (thus the persona becomes a virtual personal trainer).

* *Logging:* The rep count could be automatically logged to Wger (‚Äú10 squats done on Oct 1‚Äù). If we get even more data (like average heart rate from a smartwatch visible or from pairing with wearables), it could log intensity.

* *Challenges:* This is computationally heavy and complex to implement reliably. It might be beyond the initial scope, but it‚Äôs an exciting direction. Using Jellyfin as the platform to store and maybe trigger analysis on new videos (perhaps via a Jellyfin webhook when new media is added) could automate it. We‚Äôd have to write a custom analysis script or microservice to do the pose analysis.

* *Privacy:* All video processing would be on the local machine (perhaps requiring a GPU). None of the content should be uploaded externally. The results (like ‚ÄúGood form‚Äù or ‚ÄúIssue detected‚Äù) can be summarized and stored, but probably not the video itself (except the user keeps it in Jellyfin library if they want to review themselves).

* **6.4 Other Integrations:**

* **Smart Scales or IoT Devices:** If the user has a smart scale that logs weight to an app, we could integrate that feed (similar to wearables) to auto-update Wger weight logs.

* **Calendar Integration:** The persona could integrate with a calendar (say Google Calendar or a CalDAV local calendar) to schedule workouts or reminders. E.g., if the persona suggests a yoga session tomorrow, it could create a calendar event for it. This ties into overall life management.

* **Odoo or Other Apps:** The technical report mentioned Odoo integration for cooperative scenarios[\[48\]](https://drive.google.com/file/d/1Id3h_AbSrUmNIRw6fjucQhIyEP26rsEJ). If the user or community uses Odoo (for example to manage gym memberships or appointments), PMOVES could integrate that as well, but that‚Äôs outside our immediate scope.

* **6.5 Leveraging LangExtract Beyond Text:** We have primarily used LangExtract for document text splitting, but it could also help in extracting data from semi-structured inputs. For instance, if the nurse receives a lab report PDF for a patient, LangExtract could parse it into structured form and add it to the patient‚Äôs data pack (so the persona is aware of, say, cholesterol levels or other metrics). This broadens the medical coaching aspect when combined with Wger‚Äôs fitness data.

**Diagram: Future Integration Ecosystem** (omitted in text, but envision a diagram with wearables and camera feeding into the system). The direction is clear: we evolve from *reactive logging* to *proactive sensing*, where the system automatically captures important data and the AI personas can become even more context-aware and helpful.

---

In conclusion, this comprehensive integration plan connects PMOVES v5.12 with Wger to create a personalized wellness coaching platform. We addressed the technical compatibility (multi-agent AI meets API-first fitness data), the concrete integration steps for data syncing and persona grounding, the crucial privacy/HIPAA measures for sensitive data, and the deployment strategy using PMOVES‚Äôs existing DevOps tooling. The UX designs illustrate how two very different users ‚Äì a healthcare professional and an individual fitness enthusiast ‚Äì can benefit from AI personas grounded in their data. Finally, the optional enhancements point to a roadmap of even deeper integrations (wearables, computer vision) that could be incrementally added. This plan not only bolsters the capabilities of PMOVES (extending it into the health domain) but also showcases the power of **grounded AI personas** that can provide meaningful, context-rich assistance in daily life, all while keeping data under the user‚Äôs own control.
